# تقرير تدقيق شامل للمشروع (تقني + SEO + الأداء)

## 1) ملخص تنفيذي
- المشروع مبني على **Next.js App Router** لكنه يعتمد بشكل كبير على **Client Components** في الصفحات الأساسية (الرئيسية + الخدمات + العروض + المعرض + من نحن + التواصل)، وهذا يرفع JavaScript المطلوب للـ hydration ويؤثر على السرعة الفعلية على الجوال.
- الموقع أقرب إلى "موقع تعريفي/ثابت غني بصريًا"، لكن التنفيذ الحالي يجعل جزءًا كبيرًا منه تفاعليًا على العميل دون حاجة كاملة لذلك.
- توجد ممارسات SEO جيدة (Metadata + OG + Twitter + JSON-LD + sitemap + robots)، لكن توجد نقاط تمنع الاستفادة القصوى في الفهرسة وجودة النتائج.
- أبطأ نقطة متوقعة ليست Next.js نفسه، بل **حجم الـ JS والأنيميشن وكثافة العناصر المتحركة والصور الكثيرة داخل مكونات عميلة**.

---

## 2) فهم هيكل المشروع الحالي
- الإطار: Next.js 14 + React 18.
- نظام الصفحات: `src/app` مع `layout.tsx` و `template.tsx`.
- كل الصفحات الرئيسية تُعيد مكونات عميلة (`*Client.tsx`) بدل بناء محتوى كبير Server-first.
- `template.tsx` يلف كل الصفحات بـ `ClientLayout` (ويحتوي Navbar/Footer وWhatsApp floating button عبر motion).

**النتيجة:** حتى إن كان المحتوى ثابتًا، المتصفح ما يزال ينفذ hydration وتفاعلات كثيرة.

---

## 3) أسباب البطء الفعلية (Root Causes)

### A) تحويل معظم واجهة الموقع إلى Client Components
- الصفحة الرئيسية تعتمد مكون عميل ضخم مع Motion وتفاعلات متعددة.
- صفحة الخدمات وغيرها كذلك تعتمد مكونات عميلة كاملة.
- عندما تصبح الصفحة بالكامل عميلة، تزيد:
  - كمية JS المنزّلة.
  - وقت parse/execute.
  - وقت hydration.
  - حمل الـ main thread على الجوال.

### B) الاعتماد المكثف على مكتبة Motion في أجزاء كثيرة
- هناك استخدام واسع لـ `motion/react` في Navbar, Footer, Home, Services, PartnersSlider...
- الأنيميشن بحد ذاته ليس مشكلة دائمًا، لكن كثافته وامتداده على كامل الصفحة يسبب jank خاصة على الأجهزة المتوسطة.

### C) مكوّنات صور عميلة على نطاق واسع
- `ImageWithFallback` و`OptimizedImage` كلاهما Client Components بسبب `useState` و`onError`.
- هذا يضيف hydration حول كل موضع صورة بدل الاستفادة القصوى من Server Components + next/image مباشرة حيث ممكن.

### D) كثافة المحتوى المرئي
- مجلد الصور كبير نسبيًا (~25MB في `public/images`).
- حتى لو التحميل كسول، الصفحات الغنية بالصور + السلايدر + الأنيميشن ترفع كلفة الرسم والتمرير.

### E) مشكلة بناء تؤثر على الاستقرار
- البناء فشل بسبب جلب خطوط Google من `next/font/google` في بيئة لا تصل Google Fonts.
- في الإنتاج هذا قد يسبب هشاشة في CI/CD إن كانت البيئة مقيدة.

## 4) تقييم SEO الحالي

## نقاط قوة
- Metadata جيد على مستوى الموقع والصفحات.
- OpenGraph/Twitter مضبوطين.
- Structured Data (Organization, LocalBusiness, WebSite, FAQ, Breadcrumb, Service).
- وجود `sitemap.ts` و`robots.ts`.

## نقاط تحتاج تحسين
1. **robots.txt يمنع `/_next/`**
   - منع `/_next/` غير مستحسن غالبًا؛ Google يحتاج الوصول لملفات CSS/JS لفهم وتجربة الصفحة.
2. **عدم وجود `hreflang` حقيقي متعدد النسخ**
   - حاليًا لغة واحدة؛ جيد، لكن إن في نسخة إنجليزية مستقبلًا يلزم `hreflang` متبادل.
3. **الاعتماد على JS في عرض جزء كبير من المحتوى**
   - فهرسة Google جيدة مع JS، لكن SSR/RSC أنظف وأسرع ويعطي إشارات أفضل لسرعة الصفحات.
4. **فرص تحسين السيو المحلي (Local SEO)**
   - يلزم ربط بيانات النشاط مع Google Business Profile بدقة (NAP consistency) وإضافة صفحات مناطق/خدمات مخصصة.
5. **غياب خطة محتوى/كيورد استراتيجية**
   - الكلمات جيدة كبداية، لكن يلزم Landing Pages متخصصة حسب نية البحث (خدمة + مدينة + نوع مناسبة).

---

## 5) لماذا "سرعة Next.js" غير ظاهرة هنا؟
- Next.js سريع جدًا عندما نستفيد من:
  - Server Components افتراضيًا.
  - تقليل الـ client JS.
  - Streaming/SSR.
  - code-splitting فعّال.
- هنا حصل العكس في أجزاء كبيرة: تم دفع واجهة ضخمة إلى client (أنيميشن + state + أحداث + sliders)، فصار bottleneck في **المتصفح** وليس في Next.js.

## 6) توصيات عملية مرتبة بالأولوية

## أولوية عالية (أسبوع 1)
1. **تحويل الصفحات إلى Server-first**
   - اجعل الصفحة Server Component تعرض المحتوى الثابت.
   - انقل التفاعلات فقط إلى جزر صغيرة Client Islands (مثل سلايدر الشركاء، مودال الخدمات).
2. **تقليل Motion**
   - احذف الأنيميشن غير الضروري فوق الطية الأولى.
   - فعّل احترام `prefers-reduced-motion`.
3. **إصلاح robots**
   - لا تمنع `/_next/`.
4. **استبدال Google Fonts بتحميل محلي (next/font/local) أو استضافة ذاتية**
   - لثبات البناء وتقليل الاعتماد الخارجي.
5. **تقليل hydration في الصور**
   - استخدم `next/image` مباشرة كلما أمكن في Server Components.

## أولوية متوسطة (أسبوع 2-3)
1. **Route-level code splitting أدق**
   - lazy import للمودالات والسلايدرات الثقيلة.
2. **تقليل عدد العناصر في DOM وقت البداية**
   - خصوصًا الأقسام الطويلة في الصفحة الرئيسية.
3. **تحسين صور LCP**
   - تحديد صورة بطل واحدة priority فقط + أحجام دقيقة `sizes`.
4. **ضبط Cache-Control على مستوى CDN/Vercel**
   - التأكد من استراتيجيات cache للهتمل الديناميكي إن وجد.

## أولوية SEO نمو (مستمر)
1. **إنشاء صفحات خدمة-مدينة** (مثل: ضيافة مناسبات بالرياض، ضيافة قهوة سعودية بجدة...).
2. **إضافة قسم مقالات/معرفة** يغطي أسئلة العملاء.
3. **تحسين CTR** عبر عناوين/أوصاف أدق حسب النية.
4. **مراجعة Schema لتضمين Reviews/FAQ per page عند الاستحقاق الحقيقي**.

---

## 7) خطة قياس واضحة بعد التحسين
- راقب قبل/بعد (Lighthouse + Search Console + Core Web Vitals):
  - LCP, INP, CLS
  - JS transferred/executed
  - TTFB
  - نسبة الصفحات المفهرسة
  - CTR ومتوسط الترتيب
- المستهدف المبدئي (جوال):
  - LCP < 2.5s
  - INP < 200ms
  - CLS < 0.1

---

## 8) ملاحظات ختامية
- المشروع بصريًا جميل جدًا ومتماسك هويةً.
- المشكلة الأساسية ليست في Next.js، بل في **طريقة استخدامه** (Client-heavy).
- بمجرد التحويل إلى Server-first وتقليل الحركة والـ JS، سترى فرقًا واضحًا في السرعة ونتائج Google.
